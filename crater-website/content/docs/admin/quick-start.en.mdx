Crater Platform dependency Kubernetes The cluster is running, so before deployment, a series of basic dependent components need to be prepared. These components provide core capabilities such as monitoring, storage, networking, scheduling, and image repository, ensuring that the platform can start and run normally.

In the minimal deployment scheme, we strive to retain only the most core and indispensable dependencies and avoid introducing excessive complexity. The final determined dependencies include:

- **NVIDIA GPU Operator**：Responsible for the installation GPU Drivers, device plug-ins, and monitoring components ensure that... Crater It can be scheduled GPU Task.
- **Bitnami PostgreSQL**：Not considering high availability PostgreSQL Database services, which are mainly used in this solution. Crater external database.
- **IngressClass (Ingress-Nginx)**：Responsible for handling external traffic routing and forwarding user requests to services within the cluster.
- **Volcano**：For batch processing and AI The scheduling framework for workloads is Crater It is the core of homework scheduling.
- **StorageClass (NFS)**：A unified distributed storage backend for databases and other applications. Harbor Provide persistent storage capabilities.

The reason for choosing these components is that they are **Crater The minimum operating environment of the platform** The key support components are essential. Without them, the platform cannot function properly. For example, **Prometheus/Grafana Monitoring stack**、**MetalLB Load balancing**、**OpenEBS Storage** Dependencies that are more powerful but not essential are excluded from the minimum version to lower the deployment threshold.

## Installation

### Install from GitHub Container Registry (Recommended)

```bash
# Add the Helm OCI repository
helm registry login ghcr.io

# Install the chart
helm install crater oci://ghcr.io/raids-lab/crater --version 0.1.0

# Or upgrade an existing installation
helm upgrade crater oci://ghcr.io/raids-lab/crater --version 0.1.0
```

### Install from Source

```bash
# Clone the repository
git clone https://github.com/raids-lab/crater.git
cd crater/charts

# Install the chart
helm install crater crater/
```

## Configuration

The chart can be configured using a values file. Create a `values.yaml` file with your specific configurations:

```yaml
# Example minimal configuration
backendConfig:
  postgres:
    host: "your-postgres-host"
    password: "your-password"
  auth:
    accessTokenSecret: "your-access-token-secret"
    refreshTokenSecret: "your-refresh-token-secret"
```

Then install with your values:

```bash
helm install crater oci://ghcr.io/raids-lab/crater --values values.yaml
```

If you want to understand the detailed meaning of the configuration, you can read it here. [Configuration instructions](./chart.mdx)。

### IngressClass

In a production cluster, it's common practice to combine a load balancer (such as MetalLB）And Ingress They are used together with the controller. However, in the minimal deployment scenario, we only need a basic one. **Ingress-Nginx Controller** That's it. The external requests will be routed to the services within the cluster. This avoids the need for additional dependencies on underlying network plug-ins or load balancing functions.

You can do it directly through Helm Deploy Ingress-Nginx Controller:

```
# Add the official warehouse
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# Deploy ingress-nginx
helm install ingress-nginx ingress-nginx/ingress-nginx \
  -n ingress-nginx --create-namespace
```

### StorageClass (NFS)

NFS Provide a distributed storage backend.Harbor And CNPG Of PVC Everyone will rely on it.

1. In NFS Create a directory on the server and configure permissions:

   ```
   mkdir -p /srv/nfs/k8s
   chown -R 26:26 /srv/nfs/k8s
   ```

2. Edit `/etc/exports`，Add export rules:

   ```
   /srv/nfs/k8s 192.168.103.0/24(rw,sync,no_subtree_check,no_root_squash)
   ```

3. Refresh and export:

   ```
   exportfs -rv
   ```

4. In K8s Cluster deployment **nfs-subdir-external-provisioner**：

   ```
   helm repo add nfs-subdir-external-provisioner https://kubernetes-sigs.github.io/nfs-subdir-external-provisioner/
   helm repo update
   helm install nfs-client nfs-subdir-external-provisioner/nfs-subdir-external-provisioner \
     --namespace nfs-provisioner --create-namespace \
     --set nfs.server=192.168.103.136 \
     --set nfs.path=/srv/nfs/k8s
   ```

5. Verification StorageClass：

   ```
   kubectl get sc
   ```

   Yes, I confirm that there is `nfs-client`，And `provisioner` Yes `nfs-subdir-external-provisioner`.

### CloudNativePG (CNPG)

Kubernetes The stateful components (such as databases and image repositories) require persistent storage support. In the minimal deployment version, we choose **NFS As a unified distributed storage backend**，Because it has a simple configuration, good compatibility, and no need to deploy additional complex storage systems (such as OpenEBS、Ceph）。
 Through NFS Provided by StorageClass，Harbor And CNPG You can apply directly PVC，Achieve persistent storage without having to worry about the underlying storage details. This solution is particularly suitable for lightweight and resource-constrained environments.

CNPG As a Harbor External database management components.

1. Deploy Operator：

   ```
   helm repo add cnpg https://cloudnative-pg.github.io/charts
   helm repo update
   helm install cnpg cnpg/cloudnative-pg --namespace cnpg-system --create-namespace
   ```

2. Create a database user password Secret：

   ```
   kubectl -n cnpg-system create secret generic harbor-db-password \
     --from-literal=password='HarborDbPass!'
   ```

3. Define a database cluster (`harbor-pg.yaml`)：

   ```
   apiVersion: postgresql.cnpg.io/v1
   kind: Cluster
   metadata:
     name: harbor-pg
   spec:
     instances: 1
     imageName: swr.cn-north-4.myhuaweicloud.com/ddn-k8s/ghcr.io/cloudnative-pg/postgresql:15
     storage:
       size: 5Gi
       storageClass: nfs-client
     bootstrap:
       initdb:
         database: registry
         owner: harbor
         secret:
           name: harbor-db-password
           key: password
   ```

   ```
   kubectl apply -f harbor-pg.yaml -n cnpg-system
   ```

4. Verification CNPG：

   ```
   kubectl -n cnpg-system get pods
   kubectl -n cnpg-system get svc | grep harbor-pg
   ```

   You should see:

   - `harbor-pg-1` Pod Running
   - `harbor-pg-rw` Service Provide 5432 Port

5. Login database test:

   ```
   kubectl -n cnpg-system exec -it harbor-pg-1 -c postgres -- \
     psql -h 127.0.0.1 -U harbor -d registry -W
   ```

### Deploy Harbor

Harbor Yes Crater An image repository relied upon by the platform. In an intranet or offline environment, public image repositories (such as <zh> 平台依赖的镜像仓库。在内网或离线环境下，公共镜像库（如 <en> An image repository relied upon by the platform. In an intranet or offline environment, public image repositories (such as < DockerHub、ghcr.io）They are often inaccessible, so they must be stored in private warehouses Crater Front-end, back-end, and dependency images.
 Choose Harbor The reason is that it has **Mirror management, permission control, and security scanning** With enterprise-level capabilities, it can also be used in conjunction with Kubernetes It has good native integration. In the minimal deployment environment,Harbor You just need to rely on **NFS Storage** And **CNPG External database** It can run without any additional dependencies Redis/Postgres The built-in version simplifies the overall architecture.
 This not only ensures that Crater The mirror image can be stored and distributed locally, and it also provides a foundation for subsequent expansions (such as multi-user and multi-project mirror image management).

Harbor Use the one above. **NFS** Storage and **CNPG External database**。

1. Download Chart：

   ```
   helm repo add harbor https://helm.goharbor.io
   helm repo update
   helm pull harbor/harbor --untar --untardir ./charts
   cd charts/harbor
   ```

2. Edit `values.yaml`，Key modifications:

   - Exposure method (NodePort)：

     ```
     expose:
       type: nodePort
       nodePort:
         ports:
           http:
             port: 30002
     externalURL: http://192.168.103.136:30002
     ```

   - Use an external database:

     ```
     database:
       type: external
       external:
         host: harbor-pg-rw.cnpg-system.svc
         port: "5432"
         username: harbor
         coreDatabase: registry
         existingSecret: harbor-db-password
     ```

   - Specify PVC：

     ```
     persistence:
       enabled: true
       resourcePolicy: "keep"
       persistentVolumeClaim:
         registry:
     ```
