## I. Preliminary Background

I'm putting what I've learned into practice Rook-Ceph Previously, it was necessary for us to fundamentally understand two key concepts:**Ceph** And **Rook-Ceph**。They respectively represent the core capabilities of distributed storage systems and Kubernetes The automated operation and maintenance solution deployed on the platform.

### 1. Ceph

**Ceph It is an open-source distributed storage system.**，Originally by Sage Weil It was initiated with the aim of solving the problems of data consistency, availability, and scalability in large-scale storage environments. Unlike traditional centralized storage solutions,Ceph It does not rely on a single point controller, but is built on the basis of collaboration between peer nodes.Ceph The core design goals include:

* **High scalability**：Support translation from Chinese to English, but never translate languages other than Simplified Chinese. TB Count them PB an increase in storage capacity;
* **High availability and fault tolerance**：Ensure that the data is still available even if the node crashes, by using replication or erasure coding mechanisms;
* **Unified storage model**：At the same time, it supports block devices (Block）、Object storage (Object）And the file system (File）The visiting method.

Ceph The cluster mainly includes the following core components

| Component                         | Function description                                 |
| ---------------------------- | ---------------------------------------- |
| MON（Monitor）               | Maintain cluster status, health checks, and other metadata services such as master election |
| OSD（Object Storage Daemon） | Responsible for the core functions of data reading, writing, copying, and recovery |
| MGR（Manager）               | Provide monitoring, statistical analysis, and plug-in extension capabilities             |
| MDS（Metadata Server，(Optional) | It is used for managing the metadata of the directory tree when supporting the file system.       |

One Ceph Clusters usually require 3 More than one MON Ensure the safety of the main character. There should be multiple options OSD It is mounted on the physical disks of each node to jointly provide data storage capabilities.

### 2. Rook-Ceph

**Rook It's based on Kubernetes The storage orchestrator (Operator）**，Its goal is to**Let users manage it like they would manage their own things. Pod Manage the storage system in the same way**。Rook-Ceph It is one of the most mature and widely used backends, used to simplify Ceph In Kubernetes The deployment and management process on the platform.

Rook-Ceph Put it Ceph The deployment and maintenance are abstracted into a set. Kubernetes of the custom resources (Custom Resources），Including

| Custom resources        | Explanation                         |
| ----------------- | ---------------------------- |
| `CephCluster`     | It indicates a complete Ceph Cluster Declaration |
| `CephBlockPool`   | Management Ceph Of RBD Block storage pool    |
| `CephFilesystem`  | Management Ceph File system           |
| `CephObjectStore` | Management of object storage bucket services           |

Through Operator Mode,Rook It's done Ceph The lifecycle management of the cluster, including deployment, upgrade, fault recovery, and capacity expansion, greatly reduces the complexity of traditional management. Ceph The complexity of the installation process.

### 3. In Kubernetes Use it in Chinese Ceph

Kubernetes Native doesn't provide true persistent storage. If a Pod Bound PVC（PersistentVolumeClaim）Without a reliable back-end, it's impossible to achieve high availability, cross-node, and automatic recovery of storage capabilities. And Ceph，Especially through Rook Integrated Ceph，It just fills in the gap perfectly.

* Reliable data persistence storage;
* Elastic expansion capability;
* Support highly available block devices and shared file systems;
* Kubernetes Native support CSI Plugins and StorageClass Dynamic supply.

Rook-Ceph It is possible to build a truly usable cloud-native storage solution.
