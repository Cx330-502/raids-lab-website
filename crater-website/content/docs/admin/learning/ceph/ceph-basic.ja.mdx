## I. 前置きの背景

実践を深めていく中で Rook-Ceph 以前、2つの重要な概念を根本的に理解する必要がありました。**Ceph** そして **Rook-Ceph**。それらはそれぞれ、分散型ストレージシステムのコア機能と Kubernetes クラウド上の自動化運用・保守ソリューション。

### 1. Ceph

**Ceph これはオープンソースの分散型ストレージシステムです**，最初に Sage Weil これは、大規模なストレージ環境におけるデータの一貫性、可用性、および拡張性の問題を解決することを目的としています。従来の集中型ストレージソリューションとは異なり、Ceph 単一のコントローラに依存せず、対等なノード間のコラボレーションに基づいて構築されています。Ceph その中心的な設計目標には、以下のようなものがあります。

* **高い拡張性**：数からのサポート TB カウントダウン PB ストレージ容量の増加；
* **高可用性とフォールトトレランス**：コピーやエラー訂正コードのメカニズムにより、ノードがダウンした場合でもデータは利用可能なままになります。
* **統一ストレージモデル**：同時にブロックデバイスをサポート（Block）、オブジェクトストレージ（Object）そしてファイルシステム（File）訪問方法。

Ceph クラスターは主に以下のコアコンポーネントで構成されます

| コンポーネント                         | 機能説明                                 |
| ---------------------------- | ---------------------------------------- |
| MON（Monitor）               | クラスターの状態、健康チェック、選択主などのメタデータサービスの維持 |
| OSD（Object Storage Daemon） | データの実際の読み書き、コピー、復元などのコア機能を担当する。 |
| MGR（Manager）               | 監視、統計、プラグイン拡張機能を提供する。             |
| MDS（Metadata Server，オプション） | ファイルシステムをサポートする場合、ディレクトリツリーのメタデータを管理するために使用されます。       |

1つ Ceph クラスターは通常、必要とされます。 3 1つ以上 MON メインキャラクターの安全を確保するために、複数のキャラクターを選ぶ必要があります。 OSD 各ノードの物理ディスクにマウントされ、データストレージ機能を共同で提供します。

### 2. Rook-Ceph

**Rook それは <简体中文> に基づくものです。 Kubernetes のストレージ・オーケストレーター（Operator）**，その目標は**ユーザーに管理のようにさせる。 Pod ストレージシステムの管理も同じように行う。**。Rook-Ceph これは最も成熟した、最も広く利用されているバックエンドの1つであり、システムを簡素化するために使用されています。 Ceph で Kubernetes デプロイメントと管理プロセス。

Rook-Ceph を Ceph 配備とメンテナンスを1つのグループに抽象化する。 Kubernetes カスタムリソース（Custom Resources），含まれています

| カスタムリソース        | 説明                         |
| ----------------- | ---------------------------- |
| `CephCluster`     | 完全なものを表す Ceph クラスター宣言 |
| `CephBlockPool`   | 管理 Ceph の RBD ブロックストレージプール    |
| `CephFilesystem`  | 管理 Ceph ファイルシステム           |
| `CephObjectStore` | 管理対象のバケットサービス           |

を通じて Operator パターン、Rook 実現しました Ceph クラスターのデプロイ、アップグレード、障害復旧、スケーリングなどのライフサイクル管理により、従来のコストを大幅に削減することができます。 Ceph インストールプロセスの複雑さ。

### 3. で Kubernetes 中国で使用 Ceph

Kubernetes オリジナルは実際の永続的ストレージを提供しません。もしあなたが <en> Original doesn't provide true persistent storage. If you're using it, you're probably doing it wrong.  

（注：此处"Original"可能指某种技术或产品名称，需根据上下文判断。若为技术术语，"持久化存储"应译为"持続的ストレージ"；若为产品名称，可能需要保留英文或根据已有翻译习惯处理。） Pod バインドされた PVC（PersistentVolumeClaim）信頼できるバックエンドがなければ、高い可用性、クロスノード、自動復旧のストレージ機能は実現できません。そして。 Ceph，特に、 Rook 統合された Ceph，まさにこの空白を補うものでした。

* 信頼性の高いデータの永続的保存；
* 柔軟な拡張機能；
* 高可用性のブロックデバイスと共有ファイルシステムをサポートします。
* Kubernetes ネイティブサポート CSI プラグインと StorageClass 動的供給。

Rook-Ceph 実際に利用可能なクラウドネイティブストレージソリューションを構築することができます。

## 四、練習シーン

### 1. シーン 1: 大規模なモデルのトレーニングサービス用に独立したデータボリュームを作成する。

実験室では、<based on> をベースにしたシステムが導入されました。 PyTorch の大規模なモデルトレーニングサービスでは、各ユーザーはトレーニングコンテナにマウントされた独立のデータストレージボリュームを使用したいと考えています。管理者として、あなたはこれを実現する必要があります。

1. 名前を付けて作成してください。 `ml-training-pool` モデルのトレーニングデータ専用のブロックストレージプール。
2. このプールをバインドしてください。 Kubernetes のうちの1つ StorageClass，ユーザーが < で利用できるようにする。 PVC コールバック関数を呼び出す；
3. 作成する PVC，そして、それが正常にバインドされていることを確認してください。 Ceph クラスター内に対応するものが作成されました。 RBD ミラーリング；
4. 使用する toolbox ブロックデバイスの状態、容量などの詳細情報を確認し、その可用性を確認してください。

### 2. シーン2：シミュレートされたノードのディスク障害と Ceph データの再バランス処理

練習のために Ceph 製造環境におけるフォールトトレランスでは、あなたはそれを模擬する必要があります。 OSD ノードの異常なダウンログが発生した場合は、それを観察してください。 Ceph その自己治癒メカニズムが有効かどうか。

1. 既存のクラスターから1つを選択する。 OSD，そしてそれを < にマークします。 `out`，そのディスクをオフラインにシミュレートする。
2. 観察 Ceph データの移行、再バランス、および健全性の変化が発生したかどうか；
3. 確認してください。 OSD tree クラスターの状態変化に関して、レプリカの再構築と容量の再配分プロセスを記録します。
4. シミュレートされた故障が修復された後、それを OSD をタグ付けする `in`，システムの復旧プロセスを観察してください。

### 3. シーン3: スナップショットとロールバックを使用したトレーニングデータの保護と復元

大規模なモデルのトレーニングを開始する前に、ユーザーのトレーニングデータボリュームについて、誤削除を防ぐための復元可能なスナップショットを用意したいと思います。シミュレーションのシナリオは次のとおりです

1. 既存のブロックストレージボリューム（対応） PVC）にスナップショットを作成し、名前を付けます。 `@pretrain`；
2. ユーザーが重要なトレーニングデータを誤って削除したり、途中で異常なデータを書き込んだりした場合を想定します。
3. 使用する Ceph のスナップショットロールバック機能を使用して、それを RBD ボリュームがトレーニング前のスナップショット状態に復元されます。
4. ロールバック後のデータの一貫性を検証し、スナップショットがシステムのパフォーマンスと容量に与える影響を理解します。

### 4. シーン4: 共有ボリュームのデプロイメントで分散型トレーニングログの収集をサポートする。

実験室では、分散型トレーニング実験を実施する予定であり、複数の実験が行われる予定です。 Pod 例えば、トレーニングログは同じディレクトリに書き込む必要があります。あなたは必要です

1. 作成する Ceph ファイルシステム（CephFS），そして、 MDS サービスの提供 POSIX インターフェイス；
2. サポートを設定する ReadWriteMany の StorageClass，複数のために Pod 同時にマウントして使用する；
3. 作成する PVC，そしてそれらを複数の場所にマウントします。 Pod，シミュレーションログの並行書き込みシナリオ；
4. 書き込まれたデータが一貫しているかどうか、書き込み競合や権限異常が発生していないかどうかを検証します。

### 5. シーン5：ストレージプールの容量が尽きそうになっている場合の対応策

トレーニングタスクの提出が集中するピーク時に、あなたは何かを見つけたのです。 Ceph ブロックストレージプールの残容量が不足しそうです。あなたは必要です

1. 使用する toolbox 各々を確認してください。 Pool 使用率とコピーの設定；
2. 現在の状況を分析する。 Pool 中国にはどのようなものがあるのでしょうか。 RBD ミラーは空で、バインドされていないので、クリーンアップしてリサイクルすることを検討してください。
3. もし容量が本当に不足している場合は、いくつかの設定を調整してみてください。 Pool コピーの数を、 3 コピーを削除する 2（テストシナリオのみ）；
4. スケーリングの可能性を評価し、新しいノードを追加するか、または OSD，システムが自動的に再バランスを取るかどうかを観察してください。
